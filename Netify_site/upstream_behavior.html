Response body examples

Test 1 #1 / Test 2 #1 — ok, rate as Integer

{"rates": [{"hotel": "FloatingPointResort", "period": "Summer", "room": "SingletonRoom", "rate": 55700}]}

Bulk has 36 entries in the same format, each with "rate": Integer.

Test 1 #2 / Test 2 #4 — ok, rate as String

{"rates": [{"hotel": "FloatingPointResort", "period": "Summer", "room": "SingletonRoom", "rate": "55700"}]}

rate is a string instead of a number. HTTP status is still 200. Integer() handles both — no special case needed.

Test 1 #4 / Test 2 #3 — rate field absent from response

{"rates": [{"hotel": "FloatingPointResort", "period": "Summer", "room": "SingletonRoom"}]}
//                                                                    ^ no "rate" key

rate key is absent entirely. In bulk, all 36 entries are missing — never just a subset. This is an intermittent processing error, not quota exhaustion.

Test 1 #6 / Test 2 #5 — error body

{"message": "Failed to process rates due to an intermittent issue.", "status": "error"}
// No "rates" key at all. "status" is in the body, not the HTTP status (which is 200).

Key findings

Missing rate field is not quota exhaustion
FindingInitial design assumed a missing rate field meant quota was exhausted. Testing proved otherwise — it is an intermittent processing error appearing ~7% of the time regardless of remaining quota. Quota exhaustion has its own explicit signal: 429 with {"error":"Rate limit exceeded (1000/day)"}.
FixReclassified: HTTP 429 → QuotaExhaustedError. Missing rate field → UpstreamError. Two separate exceptions, two separate client responses.
All error patterns appear in both single and bulk requests
FindingError body and missing rate field patterns confirmed in both test runs.
Missing rate field in bulk is all-or-nothing
FindingWhen the rate field is missing in bulk, all 36 entries are affected — never just a subset. Upstream fails all-or-nothing.
Both modes show similar instability (~7% 500s, ~7% timeouts)
FindingProcessing 36 combinations does not significantly increase error rates compared to single.
Quota resets on docker compose restart rate-api
FindingConfirmed: restarting the rate-api container resets the daily counter to 0.

Special cases — upstream input validation

Tested directly against the upstream (not via our Rails app).

#ScenarioHTTP StatusBody
1No token401{"error":"Unauthorized"}
2Invalid token401{"error":"Unauthorized"}
3Invalid field value (e.g. unknown period)400{"error":"Invalid attribute: {'period': 'InvalidSeason', ...}"}
4Missing field in attribute (e.g. no period)400{"error":"Invalid attribute: {'hotel': '...', 'room': '...'}"}
5Request body {"attributes": []} — empty array200{"rates": []}
6Request body {} — attributes key missing entirely200{"rates": []}

Notes

401: empirically verified — upstream does NOT count 401s against its 1000/day quota (1000 invalid-token calls followed by a valid-token call returned 200, not 429). Our QuotaGuard consumes one unit before the HTTP call, but refunds it on 401 since the upstream rejected the request before any processing occurred.

Special cases #5 / #6: empty/missing attributes return HTTP 200 with {"rates": []}. Our controller validates params first, so this path is unreachable in practice. RateApiClient guards defensively anyway.
